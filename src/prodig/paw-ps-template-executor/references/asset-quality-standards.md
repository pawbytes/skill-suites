# Asset Quality Standards

## Overview

These standards apply to all template products, prompt packs, and digital kits. They define the minimum quality threshold for assets that customers will purchase and use.

## Universal Standards

### 1. Clarity
Every asset must be immediately understandable.

**Requirements:**
- Clear, descriptive titles (no "Untitled" or generic names)
- Consistent terminology throughout
- No unexplained jargon or acronyms
- Visual hierarchy that guides the eye

**Verification:** Someone unfamiliar with the asset should understand its purpose within 30 seconds.

### 2. Completeness
Every asset must include everything needed to achieve the promised outcome.

**Requirements:**
- All referenced files included
- All mentioned resources available
- All instructions actionable without gaps
- All placeholders clearly marked

**Verification:** Follow instructions from start to finish without needing external resources.

### 3. Consistency
Every asset must maintain uniform quality and style throughout.

**Requirements:**
- Consistent naming conventions
- Consistent formatting
- Consistent voice and tone in documentation
- Consistent visual styling (for templates)

**Verification:** Asset should feel like a cohesive product, not a collection of parts.

### 4. Usability
Every asset must work immediately with minimal friction.

**Requirements:**
- Imports/opens without errors
- Works with standard tools (no obscure requirements)
- First use succeeds without troubleshooting
- Recovery path if user makes mistakes

**Verification:** New user achieves basic outcome on first attempt.

## Type-Specific Standards

### Template Standards

| Criterion | Requirement |
|-----------|-------------|
| **Naming** | All elements clearly named, no generic placeholders |
| **Structure** | Logical organization, easy to navigate |
| **Defaults** | Sensible defaults that work for 80% use case |
| **Flexibility** | Gracefully handles customization without breaking |
| **Documentation** | Quick start under 5 minutes, full guide available |
| **Examples** | At least 2 completed examples included |

### Prompt Pack Standards

| Criterion | Requirement |
|-----------|-------------|
| **Clarity** | Each prompt has clear purpose and expected output |
| **Variables** | All inputs clearly marked with descriptions |
| **Testing** | Each prompt tested with multiple input sets |
| **Consistency** | Similar prompts follow similar structure |
| **Organization** | Logical grouping with navigation |
| **Examples** | At least 1 example output per prompt |

### Digital Kit Standards

| Criterion | Requirement |
|-----------|-------------|
| **Integration** | Components work together seamlessly |
| **Documentation** | Integration guide explains relationships |
| **Quick Start** | Results achievable in under 15 minutes |
| **Completeness** | Full workflow covered without gaps |
| **Cross-reference** | Components reference related components |
| **Value** | Total value exceeds sum of parts |

## Quality Verification Checklist

Use this checklist before finalizing any asset:

### Structural Quality
- [ ] File structure is organized and logical
- [ ] Naming is consistent throughout
- [ ] No orphaned or unused files
- [ ] All referenced files exist

### Content Quality
- [ ] Content is accurate and current
- [ ] No spelling or grammar errors in customer-facing content
- [ ] No placeholder content left in final files
- [ ] All links and references work

### Documentation Quality
- [ ] Quick start achieves promised outcome
- [ ] Full instructions are complete
- [ ] Examples are realistic and helpful
- [ ] Troubleshooting covers common issues

### Usability Quality
- [ ] First use succeeds without errors
- [ ] Error messages are helpful if they occur
- [ ] Recovery from mistakes is straightforward
- [ ] Advanced customization is possible but optional

### Value Quality
- [ ] Promised outcome is achievable
- [ ] Time-to-value meets claims
- [ ] Asset solves the stated problem
- [ ] Value justifies the price

## Rejection Criteria

Assets are rejected if they contain any of the following:

| Issue | Reason |
|-------|--------|
| Broken functionality | Asset doesn't work as intended |
| Missing components | Referenced files or resources not included |
| Thin content | Minimal effort, no real value |
| Misleading claims | Promises outcomes it can't deliver |
| Stolen content | Copied from other sources without permission |
| Malicious content | Contains harmful code, links, or instructions |

## Revision Standards

When updating assets:

### Version Numbering
- **Major (X.0)** — Breaking changes, new components
- **Minor (X.Y)** — New features, expanded content
- **Patch (X.Y.Z)** — Bug fixes, typo corrections

### Changelog Requirements
Every update must include:
- Version number
- Date
- Summary of changes
- Breaking changes (if any)
- Migration instructions (if needed)

### Backward Compatibility
- Maintain compatibility where possible
- Provide migration guides for breaking changes
- Deprecate features before removing
- Communicate changes to existing customers

## Quality Gates

### Gate 1: Functionality
Does the asset work as intended?
- All features functional
- No errors in normal use
- Edge cases handled gracefully

**Pass required before proceeding.**

### Gate 2: Documentation
Can users succeed without support?
- Quick start works for target user
- Full documentation is complete
- Examples are helpful

**Pass required before proceeding.**

### Gate 3: Value
Does the asset deliver promised outcomes?
- Outcome is achievable
- Time investment is reasonable
- Quality justifies price

**Pass required before release.**

## Post-Release Monitoring

After release, monitor for:

| Signal | Action |
|--------|--------|
| Support questions | Update documentation to address gaps |
| Feature requests | Consider for future versions |
| Negative feedback | Investigate root cause, fix if warranted |
| Positive feedback | Highlight in marketing materials |

Update assets when:
- Bug reports exceed threshold (3+ similar reports)
- Documentation gaps identified (multiple questions)
- Platform/tool changes break functionality
- New best practices emerge in the domain
