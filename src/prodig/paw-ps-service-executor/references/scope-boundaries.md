# Scope Boundaries

## Purpose

Define clear limits on what's included and excluded. Boundaries protect margins, set expectations, and make services easier to sell.

## Why Boundaries Matter

| Without Boundaries | With Boundaries |
|--------------------|-----------------|
| Scope creep erodes profit | Protected margins |
| Client expectations diverge | Aligned expectations |
| Hard to price accurately | Predictable delivery |
| Team burnout | Sustainable workloads |
| Difficult to say "no" | Clear policy reference |

## Scope Documentation Framework

### 1. Core Scope Definition

```
INCLUDED IN [Service Name]:

Direct Deliverables:
- [Deliverable 1]
- [Deliverable 2]
- [Deliverable 3]

Process Elements:
- [X] kickoff call
- [X] milestone review calls
- [X] revision rounds
- [X] async support via [channel]

Support Level:
- Response time: [X hours]
- Availability: [days/hours]
- Communication channels: [list]
```

### 2. Explicit Exclusions

```
NOT INCLUDED IN [Service Name]:

Adjacent Services:
- [Service that's separate] — available as add-on
- [Service that's separate] — available as add-on

Out of Scope Items:
- [Item client might assume is included]
- [Item client might assume is included]

Ongoing Work:
- [Maintenance/updates] — not included
- [Ongoing support beyond X] — not included

Revision Limits:
- Beyond [X] rounds: additional cost
- Scope changes: require change order
```

### 3. Boundary Lines by Service Type

**Consulting/Advisory**

| Included | Not Included |
|----------|--------------|
| Strategic recommendations | Implementation |
| Roadmap creation | Doing the work |
| Advisory calls | Unlimited access |
| Documented insights | Ongoing management |

**Done-For-You Services**

| Included | Not Included |
|----------|--------------|
| Defined deliverables | Ongoing maintenance |
| [X] revision rounds | Unlimited revisions |
| Setup/configuration | Lifetime support |
| Documentation | Training beyond [X] sessions |

**Retainer Services**

| Included | Not Included | 
|----------|-------------|
| [X] hours/month | Rollover hours |
| Defined scope | Ad-hoc requests |
| Regular check-ins | 24/7 availability |
| Listed deliverables | Out-of-scope work |

## Common Scope Creep Patterns

### Pattern 1: "Just one more thing"

```
Client: "Can you also look at [X]?"
Response: "Happy to help with [X]. That's outside our current scope.
I can add it as an additional service at $[X], or we can schedule it
for a future phase. Which would you prefer?"
```

### Pattern 2: "I assumed that was included"

```
Client: "I thought [X] was part of this."
Response: "I can see why that might seem related. To keep pricing
predictable, our scope specifically includes [list]. I can provide
a quote for [X] separately, or we can discuss adding it to the project."
```

### Pattern 3: "Small changes" accumulation

```
Client: "This is just a small change..."
Response: "I understand it seems minor. We have [X] revision rounds
included for this purpose. Let me track this change, and I'll let you
know if we're approaching the revision limit so we can prioritize."
```

### Pattern 4: Timeline extension without scope adjustment

```
Client: "We need another week to review."
Response: "Understood. Our timeline includes [X] days for client review.
Additional delays may impact the overall project schedule. Let me know
if you need an extension and we can adjust the timeline accordingly."
```

## Boundary Communication Templates

### Proposal Scope Section

```markdown
## Scope of Work

### What's Included
- [Deliverable 1]
- [Deliverable 2]
- [Deliverable 3]
- [X] revision rounds
- [X] support calls
- [Support level details]

### What's Not Included
- [Excluded item 1] — available at $[X]
- [Excluded item 2] — available at $[X]
- [Excluded item 3] — requires separate engagement

### Change Requests
Any additions or changes to the scope above will require a written
change order and may affect timeline and pricing.
```

### Agreement Scope Clause

```markdown
**Scope of Services**

Provider agrees to deliver the services described in Exhibit A
("Scope of Work"). Any services not expressly listed in Exhibit A
are excluded from this Agreement.

**Change Orders**

Client may request changes to the Scope of Work in writing.
Provider will provide a written estimate for any additional work.
Work proceeds only after Client approval of the change order.

**Client Responsibilities**

Client agrees to provide:
- [Required input 1] by [date]
- [Required feedback] within [X] days
- [Required access/assets] by [date]

Delays in Client responsibilities may affect timeline.
```

## Scope Boundary Checklist

For every service definition, document:

- [ ] Core deliverables listed explicitly
- [ ] Process elements defined (calls, revisions, support)
- [ ] Response times specified
- [ ] Exclusions listed clearly
- [ ] Change request process defined
- [ ] Client responsibilities stated
- [ ] Timeline dependencies noted

## Scope Negotiation

### When to Expand Scope

- Client needs are genuinely broader
- Expanding creates better outcome
- Client is willing to adjust investment
- You have capacity

### When to Hold Firm

- Scope would erode margins
- Sets precedent for creep
- Core deliverable would suffer
- Client expects free addition

### Negotiation Phrases

```
"That's a great addition. Let me scope that separately so we keep
the current project focused."

"I want to make sure we deliver excellence on the core scope.
Let's add [X] to a phase 2 so nothing gets diluted."

"Our agreement covers [scope]. I'm happy to discuss adding [X],
and I'll prepare a separate proposal for that."
```
