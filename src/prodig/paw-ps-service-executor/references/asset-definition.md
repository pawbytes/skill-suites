# Asset Definition

## Purpose

Specify exactly what the buyer receives — tangible deliverables and intangible outcomes that create value.

## Deliverable Categories

### Tangible Deliverables

Physical or digital assets the client can point to, share, and retain:

| Category | Examples |
|----------|----------|
| Documents | Strategy decks, roadmaps, audit reports |
| Templates | Email sequences, SOP templates, checklists |
| Creative | Designs, copy, videos, graphics |
| Technical | Built funnels, configured systems, code |
| Data | Research findings, competitor analysis, reports |

### Intangible Deliverables

Outcomes and experiences that create value:

| Category | Examples |
|----------|----------|
| Knowledge | Skills learned, insights gained |
| Access | Expert availability, network introductions |
| Confidence | Clarity on direction, decision validation |
| Time saved | Hours not spent figuring it out |
| Risk reduction | Avoided mistakes, faster path |

## Deliverable Definition Framework

For each deliverable, specify:

```markdown
## [Deliverable Name]

**Type:** [Document/Template/Creative/Technical/Data/Session]

**Format:** [PDF/Figma/Doc/Sheet/Loom video/etc.]

**Purpose:** [Why this matters to the client]

**Client Use:** [How the client uses this after delivery]

**Creation Process:**
1. [Step 1]
2. [Step 2]
3. [Step 3]

**Quality Criteria:**
- [Criterion 1]
- [Criterion 2]
- [Criterion 3]

**Dependencies:**
- [What's needed from client]
- [What must be completed first]

**Timeline:** [X days from input received]
```

## Deliverable Quality Standards

### Document Standards

```
Formatting:
- Brand-aligned (client or service provider)
- Consistent headers and hierarchy
- Page numbers on multi-page docs
- Table of contents for docs > 10 pages
- Appendix for supporting materials

Content:
- Executive summary (for docs > 5 pages)
- Clear recommendations with rationale
- Action items highlighted
- Next steps defined

File naming:
[Client]_[Deliverable]_[Version]_[Date]
Example: Acme_Strategy-Deck_v2_2024-01-15
```

### Creative Asset Standards

```
Design:
- Follows brand guidelines
- Consistent visual language
- Appropriate for intended use
- Multiple formats as needed (PNG, JPG, SVG)

Delivery:
- Organized folder structure
- Clear naming convention
- Usage notes if needed
```

### Technical Deliverable Standards

```
Functionality:
- Tested before delivery
- Documented configuration
- Access credentials provided securely
- Known limitations noted

Documentation:
- How to use
- How to maintain
- Who to contact for issues
```

## Common Deliverable Mistakes

| Mistake | Problem | Fix |
|---------|---------|-----|
| Vague deliverables | Client doesn't know what they get | Name specific files/assets |
| No formats specified | Confusion on how delivered | Always list format (PDF, Doc, etc.) |
| Missing templates | Inconsistent quality | Create templates before selling |
| Too many deliverables | Overwhelm and dilution | Focus on high-value outputs |
| No organization | Client can't find things | Provide organized folder structure |
| Missing training | Client can't use deliverables | Include how-to documentation |
