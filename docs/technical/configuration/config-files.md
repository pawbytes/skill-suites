# Configuration Files

## Location Hierarchy

Configuration files are stored in the `.pawbytes/config/` directory:

```
{project-root}/
└── .pawbytes/
    └── config/
        ├── config.yaml        # Project configuration (committed)
        └── config.user.yaml   # User overrides (gitignored)
```

### Loading Priority

Values are loaded in order, with later sources overriding earlier ones:

1. **Module defaults** — From `module.yaml` (lowest priority)
2. **Project config** — From `config.yaml`
3. **User overrides** — From `config.user.yaml` (highest priority)

---

## config.yaml

The primary project configuration file. Contains shared settings that should be version-controlled.

### File Format

```yaml
# Generated by installer
version: 1.0.0

# User identity
user_name: User
communication_language: English
document_output_language: English

# Output locations
output_folder: "{project-root}/.pawbytes/marketing-suites"
brands_folder: "{project-root}/.pawbytes/marketing-suites/brands"
reports_folder: "{project-root}/.pawbytes/marketing-suites/reports"

# Module-specific settings
default_brand: ""
```

### Standard Fields

| Field | Type | Purpose |
|-------|------|---------|
| `version` | string | Config schema version |
| `user_name` | string | User's name for personalized output |
| `communication_language` | string | Language for skill interaction |
| `document_output_language` | string | Language for generated documents |
| `output_folder` | path | Base output directory |
| `brands_folder` | path | Brand context storage |
| `reports_folder` | path | Report output location |
| `default_brand` | string | Default brand slug |

---

## config.user.yaml

User-specific overrides. This file should **not** be committed to version control as it may contain sensitive information.

### Purpose

- Store API keys and secrets
- Override personal preferences
- Keep machine-specific paths

### File Format

```yaml
# User-specific settings (not committed)
user_name: Your Name

# API Keys
fal_api_key: your-fal-api-key
pexels_api_key: your-pexels-key
elevenlabs_api_key: your-elevenlabs-key
github_token: your-github-token
```

### When to Use

| Scenario | Use config.user.yaml |
|----------|---------------------|
| API keys | Yes — sensitive data |
| Personal name | Yes — individual preference |
| Custom output paths | Maybe — if machine-specific |
| Shared team settings | No — use config.yaml |

## Variable Substitution

Configuration values can contain tokens that are resolved at runtime.

### Available Tokens

| Token | Resolves To |
|-------|-------------|
| `{project-root}` | Project root directory |
| `{user_name}` | Value of `user_name` from config |
| `{output_folder}` | Value of `output_folder` from config |
| `{module}` | Module short code (e.g., `mkt`, `cra`) |

### Example

```yaml
# In config.yaml
output_folder: "{project-root}/.pawbytes/marketing-suites"
brands_folder: "{output_folder}/brands"
```

### Important

- `{project-root}` is a **literal token** — do not substitute it with an actual path
- Skills resolve tokens at runtime
- Tokens enable portable configuration across environments

---

## Reading Configuration in Skills

Skills load configuration during the **On Activation** phase:

```markdown
## On Activation

1. Load configuration from `{project-root}/.pawbytes/config/config.yaml`
2. Load user overrides from `{project-root}/.pawbytes/config/config.user.yaml`
3. Extract relevant settings:
   - `communication_language` (default: English)
   - `document_output_language` (default: English)
   - `output_folder` for artifact paths
```

### Best Practices

1. **Always check for existing config** before prompting
2. **Use defaults** from `module.yaml` if config is missing
3. **Resolve tokens** when creating actual paths
4. **Never hardcode paths** — use configuration variables

---

## Git Ignore

Add to `.gitignore`:

```gitignore
# Pawbytes user config (contains secrets)
.pawbytes/config/config.user.yaml
```

The `config.yaml` can be committed for team sharing, but `config.user.yaml` should always be gitignored.
